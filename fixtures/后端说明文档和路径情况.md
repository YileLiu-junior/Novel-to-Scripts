## 1. 系统概览

XEngineer 是一个 **AI 小说→剧本结构化改编工作台**。后端将原始小说章节通过 4 阶段 AI 流水线转化为结构化文学剧本，并以 JSON/YAML/可读文本三种形态导出。

```
用户输入（章文本）
    → NovelReaderSkill（原著解析）
    → StoryOntologySkill（故事圣经）
    → AdaptationPlannerSkill（改编计划）
    → ScreenplayYamlWriterSkill（剧本生成）
    → SchemaValidator（结构校验）
    → ReferenceValidator（引用校验）
    → YamlExporter（YAML 导出）
    → ScreenplayRenderExporter（文学剧本渲染）
    → 7 种 Artifact 持久化
```

**核心设计原则**：
- **Schema-first**：`schemas/screenplay.schema.json` 是唯一真相源，定义所有字段类型和约束
- **代码负责结构，AI 负责内容**：角色动机、对白创作属于 AI；ID 生成、引用检查、格式校验属于后端
- **JSON 内部，YAML/Text 导出**：内部用 JSON 做校验和引用追踪；YAML 和可读文本仅作为导出格式

## 2. 环境配置

### 2.1 环境变量

所有配置通过环境变量控制，定义在 `backend/app/core/settings.py`：

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `XENGINEER_AI_PROVIDER` | `fake` | AI 提供者：`fake` / `deepseek` |
| `XENGINEER_DATA_ROOT` | `backend/data/` | 数据持久化根目录 |
| `DEEPSEEK_API_KEY` | (无) | DeepSeek API 密钥 |
| `XENGINEER_DEEPSEEK_MODEL` | `deepseek-v4-flash` | DeepSeek 模型名称 |
| `DEEPSEEK_BASE_URL` | `https://api.deepseek.com` | DeepSeek API 地址 |
| `XENGINEER_LLM_TIMEOUT_SECONDS` | `60` | LLM 调用超时（秒） |

### 2.2 启动方式

```powershell
# 使用 Fake Provider（无需 API key，本地开发/前端联调）
$env:XENGINEER_AI_PROVIDER = "fake"
uvicorn app.main:app --reload

# 使用 DeepSeek（需要 API key）
.\scripts\configure_deepseek_env.ps1    # 设置环境变量（当前终端会话）
uvicorn app.main:app --reload
```

### 2.3 Provider 模式对比

| 特性 | Fake Provider | DeepSeek Provider |
|------|--------------|-------------------|
| 需要 API Key | 否 | 是 |
| 输出内容 | 固定 fixture 数据 | AI 实时生成 |
| 输出结构 | 完全符合 Schema | 需归一化层修正 |
| 适用场景 | 前端开发、API 调试、CI | 真实剧本生成 |
| Token 消耗 | 0 | 约 8K-20K/次生成 |

## 4. 核心流程

### 4.1 生成流水线（V1 Pipeline）

```
POST /api/projects/{id}/generate/screenplay
    │
    ├─[1] 章节数校验（≥ 3 章）→ 不足则 422
    │
    ├─[2] 创建 Job（status=queued）
    │
    ├─[3] BackgroundTasks 异步执行 run_v1()：
    │       │
    │       ├─ novel_reader     → novel_analysis artifact
    │       ├─ story_ontology   → story_bible artifact
    │       ├─ adaptation_planner → adaptation_plan artifact
    │       ├─ screenplay_writer  → 生成 scenes
    │       ├─ [归一化层] 注入/修正所有必填字段
    │       ├─ [_ensure_non_empty_arrays] 保底填充
    │       ├─ ValidationService 校验
    │       ├─ 导出 YAML         → screenplay_yaml artifact
    │       ├─ 审计报告           → audit_report artifact
    │       ├─ 文学剧本渲染       → screenlay_rendered artifact
    │       └─ screenplay_json   → screenplay_json artifact
    │
    └─[4] 前端通过 GET /api/jobs/{job_id} 轮询状态
```

### 4.2 归一化层（Normalization Layer）

位于 `generation_orchestrator.py`，是连接"AI 输出"与"严格 Schema"之间的桥梁。共 12 个函数：

| 函数 | 解决的问题 |
|------|-----------|
| `_normalize_event_id` | `evt_001` / `event_1` → `event_001` |
| `_normalize_char_id` | `char_1` / `char_lin` → `char_001` |
| `_normalize_time_of_day` | 中文"夜"→ `night`，英文→小写 |
| `_normalize_interior_exterior` | 中文"内景"→ `INT` |
| `_normalize_location_payload` | 字符串 → `{name, time, interior_exterior}` |
| `_normalize_scenes` | source_refs 格式、dialogue 补 id、content_blocks 生成 |
| `_normalize_story_bible` | 角色/关系 ID 规范、缺失字段补全 |
| `_normalize_events` | event title/participants 补全 |
| `_normalize_causal_graph` | edge relation/explanation 补全 |
| `_normalize_foreshadowing` | setup_event_id 规范、缺失 ID 生成 |
| `_normalize_adaptation_plan` | scene_plan 必填字段补全 |
| `_normalize_all_references` | **全局扫描**：角色名→ID、事件名→ID、source_refs 规范化 |
| `_ensure_non_empty_arrays` | 所有 `minItems ≥ 1` 数组保底填充 + 占位场景 |

### 4.3 章节拆分流程

```
POST /api/projects/{id}/chapters/auto-split
    │
    ├─ mode=rule: 正则匹配 第X章 / Chapter X 等标记 → 切分
    ├─ mode=ai:   占位实现（后续可接 LLM 语义切分）
    └─ mode=auto: 先 rule，不足 2 章则 fallback 粗切
```

## 6. 实体与 ID 规则

所有 ID 由 `backend/app/core/ids.py` 统一生成，格式为 `{prefix}_{index:03d}`（3 位零填充）。

| 实体 | ID 格式 | 生成方 | 示例 |
|------|---------|--------|------|
| Project | `project_{shortid}` | ProjectService | `project_a1b2c3d4` |
| Chapter | `chapter_NNN` | ChapterService | `chapter_001` |
| Paragraph | `p_NNN` | ChapterService | `p_001` |
| Character | `char_NNN` | LLM 建议 + 归一化修正 | `char_001` |
| Relationship | `rel_NNN` | LLM 建议 + 归一化修正 | `rel_001` |
| Event | `event_NNN` | LLM 建议 + 归一化修正 | `event_001` |
| Foreshadowing | `foreshadow_NNN` | LLM 建议 + 归一化修正 | `foreshadow_001` |
| Scene | `scene_NNN` | LLM 建议 + 归一化修正 | `scene_001` |
| Content Block | `block_NNN` | 归一化层生成 | `block_001` |
| Dialogue Line | `line_NNN` | 归一化层生成 | `line_001` |
| Story Outline | `outline_NNN` | 归一化层生成 | `outline_001` |
| Core Action | `action_NNN` | 归一化层生成 | `action_001` |
| Situation | `situation_NNN` | 归一化层生成 | `situation_001` |
| Warning | `warning_NNN` | AuditValidator | `warning_001` |

**规则**：
- 章节/段落 ID 由代码生成，不交给 LLM
- 角色/事件/关系/伏笔 ID 可由 LLM 建议，后端归一化层负责去重和稳定
- 所有 ID 遵循正则 `^<prefix>_[0-9]{3}$`，违反此格式的值会被归一化层修正
- 禁止使用数据库自增 ID 作为公开引用标识

## 7. Schema 校验体系

### 7.1 双重校验

```
screenplay_json
    ├─ SchemaValidator（结构校验）
    │   └─ jsonschema Draft202012Validator
    │       ├─ 字段存在性（required）
    │       ├─ 类型匹配（type: string / array / object）
    │       ├─ 格式匹配（pattern: ^char_[0-9]{3}$）
    │       └─ 数量约束（minItems / minLength）
    │
    └─ ReferenceValidator（引用校验）
        ├─ scene.characters[] 必须在 story_bible.characters 中存在
        ├─ dialogue.character_id 必须在 scene.characters 中存在
        ├─ causal_graph edges 的 from/to 必须在 events 中存在
        ├─ core_elements.protagonists[] 必须在 characters 中存在
        └─ …等 14 条引用规则（详见 `docs/specs/backend-v0-v1/S6-validator.md`）
```

### 7.2 Schema 文件

- `schemas/screenplay.schema.json` — JSON Schema 2020-12，机器的唯一真相源
- 顶层 13 个必填字段：`schema_version`, `project`, `source`, `adaptation_config`, `script_structure`, `story_bible`, `core_elements`, `events`, `causal_graph`, `foreshadowing`, `adaptation_plan`, `scenes`, `audit_report`

### 7.3 ValidationFinding 结构

```json
{
  "code": "schema.invalid",
  "severity": "error",        // error | warning | info
  "message": "'title' is a required property",
  "path": "scenes.0",         // 错误路径（"."分隔）
  "target_type": "scene",
  "target_id": "scene_001"
}
```

## 8. AI Provider 与 Skill

### 8.1 Provider 架构

```
AiProvider (base.py)
    ├─ FakeProvider (fake_provider.py)
    │   └─ 从 fixtures/*.json 读取固定数据，确定性输出
    └─ DeepSeekProvider (deepseek_provider.py)
        └─ 通过 OpenAI-compatible SDK 调用 DeepSeek API
```

工厂函数 `build_ai_provider()` 在 `factory.py`，根据 `XENGINEER_AI_PROVIDER` 环境变量选择。

### 8.2 Skill 清单

| Skill | skill_name | 输入 | 输出 | Prompt |
|-------|-----------|------|------|--------|
| NovelReaderSkill | `novel_reader` | `{chapters: [...]}` | 角色候选、事件、场景、伏笔候选 | `novel_reader.md` |
| StoryOntologySkill | `story_ontology` | novel_analysis | story_bible, events, causal_graph, foreshadowing | `story_ontology.md` |
| AdaptationPlannerSkill | `adaptation_planner` | story_assets + adaptation_config | adaptation_plan（含 scene_plan） | `adaptation_planner.md` |
| ScreenplayYamlWriterSkill | `screenplay_writer` | story_assets + adaptation_plan + adaptation_config | `{scenes: [...]}` | `screenplay_writer.md` |
| ContinuityAuditorSkill | `continuity_auditor` | story_bible + events + scenes | audit_report | `continuity_auditor.md` |
| DialogueDoctorSkill | `dialogue_doctor` | scene + voice_profiles | dialogue[] | `dialogue_doctor.md` |

**注**：后两个 Skill 目前为占位，尚未接入 V1 流水线。

### 8.3 Skill 契约

- Skill wrapper 接收结构化 dict，返回结构化 dict
- Skill wrapper **不保存数据库记录**
- Skill wrapper **不决定 job state**
- Prompt 文本放在 `prompts/` 目录，代码中只引用文件名
- `ScreenplayYamlWriterSkill` 的 `schema_name = "screenplay.schema.json"` 可用于 structured output 模式

## 9. 归一化层

归一化层是"AI 非确定性输出"与"严格 Schema"之间的必要桥梁。

### 9.1 为什么需要归一化层

DeepSeek（尤其是 flash 小模型）的输出格式不稳定：
- 事件 ID 写成 `evt_001` 而非 `event_001`
- 角色引用写成名字 `"Lin"` 而非 ID `"char_001"`
- `source_refs` 写成字符串而非对象
- `dialogue` 用 `speaker` 而非 `character_id`
- 某些必填数组可能为空

归一化层在 `generation_orchestrator.py` 中作为 `screenplay_writer` 输出后的后处理步骤运行。

### 9.2 关键函数

详见上文 4.2 节。所有函数签名和用途均在 `generation_orchestrator.py` 中以中文注释标注。

### 9.3 注意事项

- 归一化是**无损**的：只补充缺失字段，不修改已存在的合法值
- 角色查找表（`_build_character_lookup`）从 `story_bible.characters` 建立 name→id 映射
- 事件查找表（`_build_event_lookup`）从 `events` 建立 title→id 映射
- `_ensure_non_empty_arrays` 是最终安全网，为所有 `minItems ≥ 1` 数组提供占位元素

## 10. 导出与渲染

### 10.1 导出产物总览

| 导出格式 | 文件类型 | 来源 | 用途 |
|----------|---------|------|------|
| `screenplay_json` | JSON | AI 生成 + 归一化 | 内部真相源、校验、引用追踪 |
| `screenplay_yaml` | YAML | YamlExporter | 人工可读的结构化剧本 |
| `screenplay_rendered` | Markdown/TXT | ScreenplayRenderExporter | 文学剧本预览/下载 |
| Schema 文档 | JSON | SchemaExporter | 供外部了解数据结构 |

### 10.2 ScreenplayRenderExporter 渲染规则

在 `exporters/screenplay_render_exporter.py` 中：

- **输入**：经过校验的 `screenplay_json`
- **输出**：Markdown 或纯文本格式的文学剧本
- **渲染逻辑**：
  - 每场以 `scene_heading.text`（如 `1. 旧巷口 外景 夜`）为标题
  - 遍历 `content_blocks` 按顺序渲染
  - `action` 块 → 段落文本
  - `dialogue` 块 → **角色名** + `action_hint`（括号注）+ 台词正文
- **不输出**：`subtext`, `surface_intent`, `emotional_state`（保持文学剧本预览干净）

    "text":     {"filename": "demo_screenplay.txt", "media_type": "text/plain; charset=utf-8", "content": "..."}

{data_root}/

    └── {project_id}/
        ├── project.json           # 项目元数据
        ├── chapters/

        └── jobs/
            └── index.json         # Job 状态

## 13. 注意事项

### 13.1 开发注意事项

1. **先跑通 Fake Provider** — 所有开发/调试先以 `XENGINEER_AI_PROVIDER=fake` 运行，确保代码逻辑正确后再切 DeepSeek
2. **API Key 绝不硬编码** — 使用 `scripts/configure_deepseek_env.ps1` 设置环境变量，该脚本已被 `.gitignore` 排除；绝不在测试文件或应用代码中写死 API key
3. **Schema 是铁律** — 任何对剧本结构的修改必须先更新 `schemas/screenplay.schema.json`，再改代码；不要反向操作
4. **归一化层要跟上** — 每次修改 Schema 时，同步检查 `generation_orchestrator.py` 的归一化函数是否需要更新
5. **ID 格式统一** — 所有 ID 必须通过 `app/core/ids.py` 的 `numbered_id()` 生成，禁止手拼字符串

### 13.2 使用 DeepSeek 注意事项

1. **DeepSeek 输出不稳定** — 每次运行结果可能不同（非确定性），同一个输入可能产生 2 个场景或 0 个场景
2. **归一化层保底** — 当 DeepSeek 输出异常时，`_ensure_non_empty_arrays` 会生成占位场景和角色，确保 pipeline 不崩溃
3. **Token 消耗** — 一次完整生成约消耗 8K-20K tokens（4 次 API 调用 × 2K-5K tokens）
4. **超时配置** — 如果模型响应慢，可调高 `XENGINEER_LLM_TIMEOUT_SECONDS`（默认 60 秒）

### 13.3 Fake Provider 注意事项

1. **Fake Provider 返回固定 fixture** — 数据来自 `fixtures/demo_screenplay.json`，用于前端开发和 API 调试
2. **Fake Provider 的 scenes 有 2 场** — 包含完整的 scene_heading、content_blocks、dialogue、character 等数据
3. **Fake Provider 不产生引用错误** — fixture 数据经过精心设计，所有 ID 引用都是闭合的

### 13.4 自动拆章注意事项

1. **支持的中文标记**：`第X章`、`第X回`、`第X卷`、`第X节`、数字序号（`1.`、`1、`）
2. **支持的英文标记**：`Chapter X`、`CHAPTER X`、`Part X`、`Book X`
3. **auto 模式**：先尝试规则匹配，不足 2 章时退化为按空行粗切
4. **拆章后自动持久化**— 调用 `auto-split` 端点会直接保存到项目，无需再调用 `PUT /chapters`

### 13.5 渲染注意事项

1. **渲染不是新的 AI step** — `ScreenplayRenderExporter` 是纯代码逻辑，按 `content_blocks` 顺序拼接已有文字
2. **不输出审查元数据** — `subtext`、`surface_intent`、`emotional_state` 默认不进文学剧本，否则预览会变成审查报告
3. **渲染依赖 schema 校验** — `render_and_save` 内部会先校验，非法 screenplay 拒绝渲染

### 13.6 常见问题

**Q: 为什么 DeepSeek 输出一直是 0 场景？**
A: 检查是否有 `_ensure_non_empty_arrays` 的占位场景被激活（artifact 中应看到 1 个 "待补场景"）。如果 DeepSeek 频繁返回空 scenes，考虑调整 `screenplay_writer.md` 的 prompt。

**Q: 为什么有 'char_lin' does not match 的错误？**
A: DeepSeek 用角色名而非 ID 作为标识。归一化层的 `_resolve_char_id` 会通过角色查找表转换，确保 `_normalize_story_bible` 给每个角色分了合法的 `char_NNN` ID。

**Q: Fake Provider 和 DeepSeek 的 artifact 数量一样吗？**
A: 是的，都是 7 种。只是数据内容不同——Fake 是 fixture，DeepSeek 是真实 AI 输出。

**Q: 如何查看某个 artifact 的具体内容？**
A: `GET /api/projects/{id}/artifacts/screenplay_json` 返回最新版；`GET /api/projects/{id}/artifacts` 列出所有。
