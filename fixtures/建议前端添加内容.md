## P0 紧急/阻塞级

### SUGGESTION-01：Scene-to-Evidence Trace（场景→改编证据溯源）

**建议力度**：P0 紧急/阻塞级

**问题**：
当前场景卡片只展示 `related_events` 和 `source_refs` 的原始 JSON，用户无法直观理解"这个场景为什么这样改编"。StoryOntology 产出的 conflict_axis、continuity_anchor、event_flow 等核心证据没有被前端消费和展示。

**产品决定**：
首版 `scene-to-evidence trace` 只放在 scene 卡片展开区，不在"改编证据"tab 中额外做跨场景索引。

**审结论原文**：
> CRITICAL GAP：StoryOntology 的用户价值还要从"证明跑了 skill"升级成"解释为什么这么改编"。demo 说服力应是：用户点开某个 scene，能追溯到对应 event、conflict_axis、source_refs、continuity_anchor。

**关联规格**：
- `01-adaptation-evidence-requirements.md` L93（AE1: schema_version + adaptation_evidence_mode 验收）
- `01-adaptation-evidence-requirements.md` L103-104（Success Criteria: 场景可追溯到 event / conflict axis / source refs / continuity anchor）
- `02-adaptation-evidence-contract.md`（前端展示子区块：完整事件、冲突轴、一致性锚点、关系与知情差、可视化表达约束、伏笔追踪）

**需添加的具体位置**：

| 位置 | 修改内容 |
|------|----------|
| `frontend/views/export.py` `_render_scenes_tab()` | 场景卡片展开区不再只展示 JSON，改为结构化 evidence trace 面板 |
| `frontend/backend_types.py` | 新增 `scene_evidence_trace(scene, screenplay) -> dict` 工具函数，从 screenplay 中关联查询 event / conflict_axis / continuity_anchor / source_refs |
| `frontend/views/export.py` | 在 `st.tabs` 中新增一个 tab：`"改编证据"`（见 SUGGESTION-03） |

**具体要求**：

1. 在场景卡片展开区（当前 `st.expander("展开查看场景正文与引用")`）内部，新增一个 **"改编证据溯源"** 子区域：
   - **关联事件**：展示 `related_events` 中每个 event 的 title、event_type、summary、event_flow（数组展开为列表）、conflict_axis（从 `story_bible.dramatic_assets.conflict_pool` 中按 event 标题模糊匹配）
   - **一致性锚点**：展示与当前场景关联的 continuity_anchors（按 `source_refs` 中 chapter_id 匹配）
   - **原文依据**：展示 `source_refs` 的人类可读形式（目前已有 `bt.ref_text()`，保持即可）
   - **冲突轴标签**：如果该场景关联的 event 对应某个 conflict（如"欲望差""信息差""立场差"），在前端打 tag 显示
   - **首版范围**：只在当前 scene 的展开区展示该 scene 的 trace，不新增全局跨场景索引或独立 trace 列表。

2. 用户视角的交互流：
   ```
   点开场景卡片 → 看到 scene_heading + 角色 + 对白
   → 展开"改编证据溯源"
   → 看到：
        📌 关联事件：event_001 初遇（narrative）→ 信息差
        🔗 原文依据：chapter_001 p_003
        ⚓ 一致性锚点：[称呼规范] 男主必须首次亮相时使用全名
        🎭 冲突轴：欲望差 / 信息差
   ```

3. **不需要新增后端接口**：所需数据已在 `screenplay_json` artifact 中。前端从现有 `screenplay_data` 中提取 `events`、`story_bible.dramatic_assets.conflict_pool`、`story_bible.continuity_anchors` 即可。

**需对接的接口**：
- 已有 `frontend/api_client.py` `get_artifact(project_id, "screenplay_json")` — 数据源不变化

**理由**：
这是 Demo 说服力的核心差距。当前前端能"展示数据"，但不能"解释改编"。如果没有 scene-to-evidence trace，StoryOntology 的存在对用户而言只是 backend 的工程检查点，用户无法感知其价值。ACE（Acceptance Example）要求 demo 必须能走通 scene → event → conflict_axis → source_refs 的完整追溯。

## P1 高优先级/严重影响业务流程

### SUGGESTION-02：隐藏 minimal 模式，不暴露为前端开关

**建议力度**：P1 高优先级

**问题**：
`adaptation_evidence_mode` 的 `minimal` 值是 backend/debug/legacy 兼容用途，不应作为前端可选的用户配置项。如果前端暴露 "生成模式：完整 / 最小" 的开关，会削弱"改编证据层是核心价值"的产品叙事。

**审结论原文**：
> minimal 模式不要做成用户显眼开关。产品上我会把 minimal 明确写成 backend/debug/legacy compatibility only，前端默认不暴露，否则会削弱"改编证据层是核心价值"的叙事。

**关联规格**：
- `02-adaptation-evidence-contract.md` L29-33（`minimal` 用于兼容旧 demo 或定位问题）
- `04-storyontology-skill-prompt-reference.md` L195-206（`minimal` 只输出 legacy shape）

**需添加的具体位置**：

| 位置 | 修改内容 |
|------|----------|
| `frontend/views/original.py` | 生成配置区**不添加** `adaptation_evidence_mode` 下拉框/radio。生成时 `adaptation_config` 始终传入 `"adaptation_evidence_mode": "enabled"` 或不传（后端默认 enabled） |

**具体要求**：

1. **不添加** `adaptation_evidence_mode` 选择器。前端 `generate_screenplay()` 调用中，`adaptation_config` 要么不包含此字段（让后端用默认值 `enabled`），要么硬编码为 `"enabled"`。
2. 如需 debug 入口（内部开发者使用），可考虑：
   - 在 `api_client.py` 中保留参数传递能力
   - 在 UI 中通过 Streamlit `st.query_params`（URL 参数 `?debug=1`）或环境变量控制，不显示为正常用户的可见开关

**需对接的接口**：
- `POST /api/projects/{project_id}/generate/screenplay` — `adaptation_config` 字段中不传或传 `"adaptation_evidence_mode": "enabled"`

**理由**：
产品定位：改编证据层是 XEngineer 区别于"一句 prompt 出剧本"的核心差异化能力。如果前端给用户一个 "完整模式 / 最小模式" 开关，用户会困惑"我为什么需要最小模式"，且暗示改编证据是可选的锦上添花，而非核心价值。

### SUGGESTION-03：统一产品命名为"改编证据"

**建议力度**：P1 高优先级

**问题**：
负责人审之后，规格文档已收束对外主命名为"改编证据"。前端 UI 中 story_bible 相关内容仍散落在"角色与关系"和"情节与因果"两个 tab 中，缺少一个统一的"改编证据"入口。`02-adaptation-evidence-contract.md` 已规定主 tab 应叫 "改编证据"。

**审结论原文**：
> 产品命名要收束。建议对外统一主名：改编证据。story_bible 保留为 artifact/internal term，不作为主要 UI 文案。

**关联规格**：
- `01-adaptation-evidence-requirements.md` L62-64（R3: 文案避免把 ontology 作为主界面术语）
- `02-adaptation-evidence-contract.md`（主 tab：改编证据；子区块：完整事件、冲突轴、一致性锚点、关系与知情差、可视化表达约束、伏笔追踪）

**需添加的具体位置**：

| 位置 | 修改内容 |
|------|----------|
| `frontend/views/export.py` L371 `st.tabs()` | 在现有 6 个 tab 中新增一个 tab：`"改编证据"` |
| `frontend/views/export.py` | 新增 `_render_evidence_tab(screenplay)` 函数 |
| `frontend/backend_types.py` | 新增 `conflict_pool`、`continuity_anchors`、`filmic_constraints` 相关的提取函数 |

**具体要求**：

1. 新增 `"改编证据"` tab，内部子区块：

   | 子区块 | 标题文案 | 数据来源 | 展示方式 |
   |--------|----------|----------|----------|
   | 完整事件 | `完整事件（N个）` | `screenplay.events` | 时间线列表 + event_flow 展开 + conflict tag |
   | 冲突轴 | `冲突轴` | `story_bible.dramatic_assets.conflict_pool` | 按类型（欲望差/信息差/立场差/关系压力/时间压力/能力身份差）分组卡片 |
   | 一致性锚点 | `一致性锚点` | `story_bible.continuity_anchors` | 按类别（称呼规范/人设固定点/关键道具/时间线/世界规则）分组列表 |
   | 关系与知情差 | `关系与知情差` | `story_bible.relationship_edges` + `story_bible.knowledge_states` | 关系表 + 知情状态 |
   | 可视化表达约束 | `可视化表达约束` | `story_bible.dramatic_assets.filmic_constraints` | 清单列表（内心→动作/旁白→视觉/道具空间/角色首亮相锚点） |
   | 伏笔追踪 | `伏笔追踪` | `screenplay.foreshadowing` | 现状已在"情节与因果"tab，可在此复用或链接 |

2. **角色与关系 tab 保持不变**（角色卡片作为独立 tab 仍有价值）。但"改编证据"tab 中的"关系与知情差"聚焦于证据层（关系状态+知情差），而非角色卡片。

3. **前端所有 UI 文案**中：
   - 使用 `改编证据`（对外主名）
   - 使用 `story_bible` 仅限代码标识符、API 字段名、artifact type（内部 term）
   - 使用 `可视化表达约束` 替代 `filmic_constraints`（见 SUGGESTION-05）

**需对接的接口**：
- 已有 `frontend/api_client.py` `get_artifact(project_id, "screenplay_json")` — 数据源不变化
- 已有 `frontend/api_client.py` `get_artifact(project_id, "story_bible")` — 作为 artifact 降级时的备用数据源

**理由**：
产品命名的统一直接影响用户对产品价值的理解。"改编证据"传达了"AI 的产出不是黑盒剧本，而是可审查的改编依据"这一核心定位。UI 散落会导致用户困惑"故事圣经是什么？和情节与因果有什么区别？"

### SUGGESTION-04：Planner 消费点可视化 — must_keep_together 事件状态

**建议力度**：P1 高优先级

**问题**：
当前前端只展示 `adaptation_plan` 的 JSON（`_render_plot_tab()` 中 `st.json(plan.get("scene_plan", []))`），用户无法直观理解 planner 对 StoryOntology 产出的消费情况。特别是标记为 `must_keep_together` 的完整事件，如果只落到 audit warning，用户体感弱。

**审结论原文**：
> planner 消费点要被用户看见。如果只落到 audit warning，用户体感仍弱。建议在前端 evidence 区给完整事件加状态，比如"已保护 / 被拆分需说明 / 未进入 scene plan"，这会明显提升 StoryOntology 的存在感。

**关联规格**：
- `03-implementation-plan.md` L73-81（D4: 最小 planner 消费只做完整事件保护）
- `01-adaptation-evidence-requirements.md` L81-82（R11: must_keep_together event 不能在 scene plan 中被无依据拆散）

**需添加的具体位置**：

| 位置 | 修改内容 |
|------|----------|
| `frontend/views/export.py` `_render_evidence_tab()`（新增）| 在"完整事件"子区块中，每个事件卡片显示 planner 消费状态 tag |
| `frontend/backend_types.py` | 新增 `event_planner_status(event, adaptation_plan) -> str` 辅助函数 |

**具体要求**：

1. 在"改编证据 → 完整事件"中，每个事件卡片显示状态 tag：

   | 状态 | 文案 | 颜色/样式 | 判断条件 |
   |------|------|-----------|----------|
   | 已保护 | `已保护` | 🟢 绿色 | event id 出现在 `adaptation_plan.scene_plan[*].source_events` 中，且事件被完整保留（未被拆分到多个 scene） |
   | 被拆分需说明 | `被拆分需说明` | 🟡 黄色 | event id 出现在多个 scene 的 `source_events` 中（被拆散），且 event 在原 `events[]` 中标记为 `must_keep_together` |
   | 未进入 scene plan | `未关联场景` | ⚪ 灰色 | event id 未出现在任何 `scene_plan[*].source_events` 中 |

2. 如果存在"被拆分需说明"或"未关联场景"的事件，在该区块顶部显示汇总提示：
   ```
   ⚠️ N 个标记为不可拆分的完整事件在改编计划中被拆散或遗漏。请在"审查报告"中查看详情。
   ```

3. `must_keep_together` 判断逻辑（`backend_types.py`）：
   - 从 `screenplay.events[*]` 中检查 event 是否有 `must_keep_together: true` 字段
   - 在 `adaptation_plan.scene_plan[*].source_events` 中查找 event id 的出现次数

**需对接的接口**：
- 已有 `frontend/api_client.py` `get_artifact(project_id, "screenplay_json")` — events + adaptation_plan 在同一 artifact 中

**理由**：
这是 StoryOntology 价值的"最后一公里"。如果 StoryOntology 产出了完整事件和 must_keep_together 标记，但前端没有展示 planner 是否尊重了这些标记，用户看到的仍然是"AI 自动生成了剧本"，而不是"AI 在改编证据约束下生成了剧本"。状态 tag 是 StoryOntology 从"后端工程检查"变成"用户可感知的改编保障"的关键。

## P2 用户体验升级

### SUGGESTION-05：filmic_constraints 文案收紧为"可视化表达约束"

**建议力度**：P2 次要/用户体验升级

**问题**：
`filmic_constraints` 容易被误解为镜头设计或拍摄指导。当前规格已明确边界（不写镜头设计、分镜或视频 prompt），面向用户的中文展示名应统一为"可视化表达约束"。

**审结论原文**：
> filmic_constraints 容易被误解成镜头设计。建议文案再收紧成"可视化表达约束"，强调它只回答"哪些小说信息必须转成可演、可见、可听的表达"，不回答"怎么拍"。

**关联规格**：
- `04-storyontology-skill-prompt-reference.md` L273-282（filmic_constraints 边界：不写镜头设计、分镜或视频 prompt）
- `02-adaptation-evidence-contract.md`（子区块使用"可视化表达约束"）

**需添加的具体位置**：

| 位置 | 修改内容 |
|------|----------|
| `frontend/views/export.py` `_render_evidence_tab()`（新增）| 子区块标题使用 `可视化表达约束` |
| `frontend/views/export.py` | 在该区块顶部添加一行说明文案 |

**具体要求**：

1. 前端 UI 中所有面向用户的文案使用 `可视化表达约束`，不使用 `filmic_constraints` 或 `影视化约束`。
2. 在"可视化表达约束"子区块顶部添加说明：
   ```
   💡 以下列出了小说信息中必须转化为可演、可见、可听表达的内容。这不是拍摄指导或镜头设计。
   ```
3. 展示方式：
   - 按类别分组（内心→动作/旁白→视觉/道具空间/角色首亮相锚点）
   - 每条显示 `description` + `source_refs`
   - 简洁清单样式，非 JSON dump

**需对接的接口**：
- 已有接口，数据从 `story_bible.dramatic_assets.filmic_constraints` 获取

**理由**：
产品文案精度影响用户信任。如果用户看到"影视化约束"而看到的是"内心状态需要转为动作/台词"，不熟悉边界的用户可能觉得"不够专业"（为什么没有镜头建议？），而熟悉影视的用户可能担心"越界"（AI 不该建议怎么拍）。收紧文案可以精确管理预期。
