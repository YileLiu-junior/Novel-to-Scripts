## 2. 接口总览（含完整 URL）

> **Base URL**：开发环境默认 `http://localhost:8000`

### 2.1 项目 Project

| 方法 | URL | 说明 | 请求体 | 响应关键字段 |
|------|-----|------|--------|-------------|
| `POST` | `/api/projects` | 创建项目 | `{title, logline?, target_format?}` | `id, title, logline, target_format` |
| `GET` | `/api/projects/{project_id}` | 获取项目详情 | — | `id, title, logline, target_format` |

### 2.2 章节 Chapter

| 方法 | URL | 说明 | 请求体 | 响应关键字段 |
|------|-----|------|--------|-------------|
| `PUT` | `/api/projects/{project_id}/chapters` | 导入/覆盖章节 | `{chapters: [{title, text}]}` | `[{id, order, title, paragraphs}]` |
| `GET` | `/api/projects/{project_id}/chapters` | 获取已保存章节 | — | `[{id, order, title, paragraphs}]` |
| `POST` | `/api/projects/{project_id}/chapters/auto-split` | 粘贴全文自动拆章 | `{text, mode?}` | `{chapters, chapter_count, mode_used}` |

`auto-split` 会自动忽略声明、广告、目录和序章，默认只保存正文第 1-3 章。响应形状不变；如果后端调用 AI boundary planner，调试 trace 会保存为 `chapter_split_plan` artifact。

### 2.3 生成 Generation

| 方法 | URL | 说明 | 请求体 | 响应关键字段 |
|------|-----|------|--------|-------------|
| `POST` | `/api/projects/{project_id}/generate/screenplay` | **触发全流水线** | `{adaptation_config?}` | `{job_id, status}` |

**adaptation_config 完整字段（全部可选，有默认值）**：
```json
{
  "adaptation_config": {
    "target_format": "web_series",           // web_series | short_drama | film | general
    "fidelity_level": "high",                // low | medium | high
    "preserve_priorities": ["relationship_arc"],  // 保留重点
    "dialogue_style": "restrained_with_subtext"   // 对话风格
  }
}
```

### 2.4 任务 Job

| 方法 | URL | 说明 | 响应关键字段 |
|------|-----|------|-------------|
| `GET` | `/api/jobs/{job_id}` | **轮询任务状态** | `id, project_id, status, current_step, error, artifact_ids` |

**status 流转**：`queued → running → succeeded / failed`
**current_step 枚举**：`novel_reader | story_ontology | adaptation_planner | screenplay_writer | complete`

前端轮询逻辑：
```
POST /generate → 拿到 job_id
  ↓
每隔 1.5-2s GET /api/jobs/{job_id}
  ↓
status === "succeeded" → 跳转结果页
status === "failed"    → 显示 error 信息
```

### 2.5 产物 Artifact

| 方法 | URL | 说明 |
|------|-----|------|
| `GET` | `/api/projects/{project_id}/artifacts` | 列出全部 artifact 摘要 |
| `GET` | `/api/projects/{project_id}/artifacts/{type}` | 获取某类型最新完整数据 |

**可用的 type 值**：
```
novel_analysis   → 原著解析结果（角色候选、事件、场景候选）
chapter_split_plan → 原始全文拆章 trace（ignored spans、AI boundary plan、warnings）
story_bible      → 故事圣经（角色、关系、知识状态）
adaptation_plan  → 改编计划（保留/合并/删除事件 + 场景规划）
screenplay_json  → **结构化剧本 JSON（前端主要消费的数据源）**
screenplay_yaml  → YAML 剧本（字符串）
screenplay_rendered → 文学剧本文本（Markdown/TXT）
audit_report     → 审查报告（警告列表）
```

### 2.6 YAML 剧本

| 方法 | URL | 说明 |
|------|-----|------|
| `POST` | `/api/projects/{project_id}/yaml/validate` | 校验用户编辑后的 YAML |
| `GET` | `/api/projects/{project_id}/yaml/download` | 下载 YAML 文件 |

### 2.7 文学剧本预览

| 方法 | URL | 说明 |
|------|-----|------|
| `GET` | `/api/projects/{project_id}/screenplay/rendered?format=markdown` | 预览 Markdown 文学剧本 |
| `GET` | `/api/projects/{project_id}/screenplay/rendered?format=text` | 预览纯文本文学剧本 |
| `GET` | `/api/projects/{project_id}/screenplay/rendered/download?format=markdown` | 下载 .md 文件 |
| `GET` | `/api/projects/{project_id}/screenplay/rendered/download?format=text` | 下载 .txt 文件 |

### 2.8 Schema 文档

| 方法 | URL | 说明 |
|------|-----|------|
| `GET` | `/api/projects/{project_id}/schema/download` | 下载 Schema 说明文档 |

```json
  "voice_profile": {"rhythm": "...", "diction": "..."},
```

### 3.4 Artifact 通用结构

```json
{
  "id": "artifact_project_xxx_screenplay_json_v001",
  "type": "screenplay_json",
  "version": 1,
  "data": { ... }  // 具体内容，不同类型 data 结构不同
```

- `screenplay_yaml` 的 `data` 是**字符串**
- 其余类型（JSON artifact）的 `data` 是 **JSON object**

## 4. 核心流程与前端交互

### 4.1 完整用户流程

```
[导入页面]
  ├─ 方式 A：手动输入章节
  │   └─ PUT /api/projects/{id}/chapters
  │       每章 {title, text}
  │
  ├─ 方式 B：粘贴全文自动拆分
  │   └─ POST /api/projects/{id}/chapters/auto-split
  │       {text, mode: "auto"}
  │
  └─ 方式 C：文件上传（前端自行解析为 text，再用方式 A 或 B）

[生成确认页]
  ├─ GET /api/projects/{id}/chapters → 确认章节数 ≥ 3
  └─ POST /api/projects/{id}/generate/screenplay → 触发生成

[进度轮询]
  └─ GET /api/jobs/{job_id}（每 1.5-2s 轮询）
      ├─ status=queued    → "排队中"
      ├─ status=running   → 显示 current_step（小说解析 / 故事圣经 / 改编计划 / 剧本生成）
      ├─ status=succeeded → 跳转结果页
      └─ status=failed    → 显示 error + 重试按钮

[结果展示页]
  ├─ YAML 结构预览 Tab
  │   ├─ GET /api/projects/{id}/artifacts/screenplay_yaml → 原始 YAML
  │   └─ GET /api/projects/{id}/yaml/download → 下载 .yaml
  │
  ├─ 文学剧本预览 Tab
  │   ├─ GET /api/projects/{id}/screenplay/rendered?format=markdown → 预览
  │   └─ GET /api/projects/{id}/screenplay/rendered/download?format=markdown → 下载 .md
  │
  ├─ 结构化数据 Tab
  │   ├─ 场景列表 → screenplay_json.scenes[]
  │   │   └─ 每场：scene_heading.text, characters, dramatic_purpose, dialogue
  │   ├─ 角色卡片 → story_bible.characters[]
  │   │   └─ 每个角色：name, narrative_role, voice_profile, source_refs
  │   ├─ 事件图谱 → events[] + causal_graph.edges[]
  │   └─ 改编计划 → adaptation_plan（retained/merged/deleted/protected）
  │
  ├─ 审查报告 Tab
  │   └─ audit_report（continuity_warnings / dialogue_warnings / schema_warnings）
  │
  └─ 伏笔追踪 Tab
      └─ foreshadowing[]（setup / payoff / status）
```

### 4.2 前端状态管理建议

```typescript
// 项目状态

  projectId: string;

// 生成后消费的数据

```

## 5. Schema 真相源

### 5.1 什么是 Schema

`schemas/screenplay.schema.json` 是所有数据结构的**唯一真相源**。它定义了：
- 每个字段的类型（string / number / array / object）
- 哪些字段是必填的（required）
- ID 格式约束（pattern: `^char_[0-9]{3}$`）
- 数组最小长度（minItems）
- 枚举值（enum: `["day", "night", "morning", "dusk"]`）

### 5.2 前端如何使用

1. **查阅字段定义**：打开 `schemas/screenplay.schema.json`，在 `properties` 中找字段名 → 看 `type`、`required`、`description`
2. **查阅子类型**：在 `$defs` 中查 `scene`、`character`、`dialogueLine` 等的精确字段
3. **不要自己去猜字段名** — schema 是单源真相，后端保证所有返回数据符合 schema

### 5.3 示例 fixture（前端 mock 数据）

| Fixture | 路径 | 用途 |
|---------|------|------|
| 3 章示例原文 | `fixtures/demo_novel_3_chapters.json` | 导入页 mock |
| 示例故事圣经 | `fixtures/demo_story_bible.json` | 角色页 mock |
| 示例完整剧本 | `fixtures/demo_screenplay.json` | 剧本展示页 mock |

这三个 fixture 的数据全部合法（能通过 schema 校验 + 引用校验），前端可以用它们做独立开发，完全不需要跑后端。

## 6. Fake Provider：前端无需接真实 AI

### 6.1 原理

后端有两种运行模式：

| 模式 | 环境变量 | 行为 |
|------|---------|------|
| **Fake**（默认） | `XENGINEER_AI_PROVIDER=fake` | 调用 4 个 Skill 时直接返回 `fixtures/demo_screenplay.json` 的固定数据 |
| **DeepSeek** | `XENGINEER_AI_PROVIDER=deepseek` | 真实调用 AI，耗时 30-90 秒，需 API key |

### 6.2 前端开发建议

1. **全程用 Fake 模式**：`uvicorn app.main:app`（默认就是 fake）
2. 创建项目 → 导入 3 章 → 触发生成 → 约 1 秒后 job 完成
3. 所有 API 返回的都是真实结构的 fixture 数据
4. 前端调通所有页面后，再切 DeepSeek 验证

### 6.3 Fake 模式数据特征

- 2 场完整的 scene（贝克街合租、案子敲门）
- 3 个角色（华生、福尔摩斯、斯坦福德）
- 多个事件 + 伏笔 + 审查警告
- 所有 ID 引用全部闭合（角色在场景中，事件在因果图中等）

## 7. 数据持久化与状态管理

### 7.1 数据存储位置

所有数据存在 `XENGINEER_DATA_ROOT` 环境变量指定的目录（默认 `backend/data/`），结构如下：

```
data/projects/{project_id}/
├── project.json
├── chapters/index.json
├── artifacts/
│   ├── index.json
│   ├── novel_analysis_v001.json
│   ├── story_bible_v001.json
│   ├── adaptation_plan_v001.json
│   ├── screenplay_json_v001.json
│   ├── screenplay_yaml_v001.yaml
│   ├── screenlay_rendered_v001.json
│   └── audit_report_v001.json
└── jobs/index.json
```

### 7.2 版本管理

- 每次触发生成，所有 artifact 版本递增（v001 → v002 → v003）
- 旧版本不会被删除
- `GET /artifacts/{type}` 默认返回**最新版本**
- 目前没有历史版本浏览 API（V2 计划）

### 7.3 数据重置

```
# 删除所有数据（从头开始）
rm -rf backend/data/projects/

# 改数据根目录（隔离测试）
$env:XENGINEER_DATA_ROOT = "C:\Temp\xengineer-test"
```

## 8. 页面/专栏建议

### 8.1 推荐页面结构

```
📁 项目列表
  └─ 创建新项目 / 打开已有项目
      │
      ├─ 📝 导入页（Chapter Input）
      │   ├─ 手动输入：标题 + 正文
      │   ├─ 粘贴全文：自动拆分（调用 auto-split）
      │   └─ 文件上传（前端解析为 text）
      │
      ├─ ⚙️ 改编设置（Adaptation Config）
      │   ├─ 目标格式（web_series / short_drama / film）
      │   ├─ 忠实度（low / medium / high）
      │   └─ 对话风格
      │
      ├─ ▶️ 生成（触发生成 + 进度轮询）
      │
      └─ 📊 结果展示（Tabs）
          ├─ Tab 1: 🎬 文学剧本预览
          │   └─ 调用 /screenplay/rendered（Markdown 渲染）
          │
          ├─ Tab 2: 📋 YAML 结构
          │   └─ mono 字体展示 YAML 原文 + 下载按钮
          │
          ├─ Tab 3: 🎭 角色与关系
          │   ├─ 角色卡片（name, narrative_role, voice_profile）
          │   ├─ 关系图（from → to, type, current_state）
          │   └─ 知识状态表（谁知道什么）
          │
          ├─ Tab 4: 🗺️ 场景编排
          │   ├─ 场景列表（id, title, location, characters, dramatic_purpose）
          │   ├─ 展开单场：content_blocks 自然段 + dialogue 列表
          │   └─ source_refs 追溯（点击跳转到原文章节）
          │
          ├─ Tab 5: 📈 情节与因果
          │   ├─ 事件时间线（events[] 按因果关系排列）
          │   ├─ 因果图（causal_graph.edges）
          │   └─ 伏笔追踪（foreshadowing status: open / paid_off）
          │
          └─ Tab 6: 🔍 审查报告
              ├─ schema_warnings（结构问题）
              ├─ continuity_warnings（连贯性问题）
              ├─ dialogue_warnings（对白问题）
              └─ unresolved_foreshadowing（未兑现伏笔）
```

## 9. 错误处理

### 9.1 HTTP 状态码

| 状态码 | 场景 | 响应示例 |
|--------|------|---------|
| `200` | 成功 | — |
| `404` | 资源不存在 | `{"detail": "Project not found."}` |
| `404` | 无 rendered artifact | `{"detail": "No screenlay_rendered artifact. Run generation first."}` |
| `404` | 无 YAML artifact | `{"detail": "No screenplay_yaml artifact exists for this project yet."}` |
| `422` | 章节不足 3 章 | `{"detail": {"code": "cannot-generate", "message": "至少需要 3 章"}}` |
| `400` | 不支持的 format | `{"detail": "Unsupported format: pdf. Use 'markdown' or 'text'."}` |

### 9.2 Job 失败处理

```typescript
if (job.status === 'failed') {
  // job.error 包含具体错误信息，例如：
  // "Cannot export invalid screenplay: 'title' is a required property"
  // "Core elements reference missing protagonist char_001."
  // 显示给用户：生成失败 + error 信息 + 重试按钮
}
```

**常见失败原因**：
- DeepSeek 模型输出格式异常（概率约 10-20%）
- 归一化层已处理大部分情况，极端时会生成占位数据
- 重试通常能解决（DeepSeek 的非确定性意味着下次可能成功）
