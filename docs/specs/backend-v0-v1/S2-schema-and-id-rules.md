# S2 Schema And ID Rules

## Owner

Contract Architect with Validation Director.

## Purpose

Define the machine-readable screenplay shape and stable ID rules that make
references checkable.

## Files

- `schemas/screenplay.schema.json`
- `schemas/screenplay.schema.yaml`
- `schemas/screenplay-schema-design.md`
- `backend/app/core/ids.py`
- `docs/schema/screenplay-schema-explained.md`

## ID Rules

- Chapter IDs are assigned by backend order: `chapter_001`, `chapter_002`.
- Paragraph IDs are stable inside a chapter: `p_001`, `p_002`.
- Scene and line IDs are generated by screenplay generation or normalization:
  `scene_001`, `line_001`.
- Warning IDs are assigned by audit mapping: `warning_001`.
- Model-suggested character, event, relationship, and foreshadowing IDs may be
  normalized by backend code before persistence.
- Do not use database auto-increment IDs as public references inside YAML.

## Schema Rules

- Top-level required fields:
  - `schema_version`
  - `project`
  - `source`
  - `adaptation_config`
  - `story_bible`
  - `events`
  - `causal_graph`
  - `foreshadowing`
  - `adaptation_plan`
  - `scenes`
  - `audit_report`
- JSON schema is canonical; YAML schema mirrors it for reader convenience.
- Schema checks field presence and types. Cross-reference checks live in
  `backend/app/validators/reference_validator.py`.

## Acceptance

- `demo_screenplay.json` validates against JSON schema shape.
- `demo_screenplay.yaml` mirrors the same structure.
- Broken references are not hidden by schema; they are surfaced by validators.
