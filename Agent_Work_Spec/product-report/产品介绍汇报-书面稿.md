# 改编室：AI 小说转 YAML 剧本工作台 —— 产品介绍报告

**产品定位**：用轻量故事本体守住原著因果、伏笔与人物声音的 AI 改编工作台

---

## 一、问题：AI 转剧本，为什么创作者不买账？

市面上已有的 AI 小说转剧本工具，底层逻辑几乎一致：

```
小说文本 → 一个 Prompt → 剧本文本
```

这是一条**黑箱路径**。创作者把小说丢进去，拿到一段"看起来像剧本"的输出，但真正用它工作的人——编剧、小说作者——很快就会发现五个让他们无法忍受的问题：

### 痛点 1：缺乏"潜台词（Subtext）"的转译

小说写"他看着她，笑了笑，把手抽了回来。"AI 转剧本时往往直接写台词："我不喜欢你了，我要走。"——**把潜台词直接说穿**，彻底丧失了戏剧的含蓄美。编剧拿到这样的剧本，每一句台词都要重写。

### 痛点 2：台词"千人一面"，语法正确式的平庸

AI 生成的台词普遍像个"情绪稳定的当代大学生"。黑帮老大、5 岁萌娃、没上过学的旧社会老太太——**所有人说话都语法正确、句式工整**。人物的风骨、阶层黑话、地域方言完全丧失。这是编剧最不能容忍的"AI 感"。

### 痛点 3：修改时的"全局坍塌"灾难

剧本创作是网状的。编剧写到第 50 场戏时想回去第 5 场埋一个伏笔，或者制片人要求"把女二号的职业从医生改成律师"。如果用普通 AI 工具——改了第 5 场，后面 45 场无法协同修改；而一旦让 AI 重新全量生成，**之前人工润色好的台词全部被随机性覆盖**。这就是创作者口中的"黑盒回溯灾难"。

### 痛点 4：伏笔和因果链在摘要过程中被吞噬

百万字小说进 AI，第一步往往是摘要压缩。在这个过程中，原著里精心埋设的小伏笔、因果暗线——"第三章那个茶杯的特写其实暗示了第七章的投毒"——**在语义蒸馏阶段就被当作噪声过滤掉了**。

### 痛点 5：改编尺度的"非黑即白"

要么像个"复印机"，把小说对话一字不差搬过来，剧本极其拖沓毫无视听感；要么一开"创造力"就彻底放飞自我，把原著核心主线和名场面全给魔改了。**创作者无法控制 AI 的改编尺度**。

---

## 二、洞察：这些痛点的根因是什么？

回顾上面的五个问题，它们的共同根因不是"AI 不够强"，而是一个**架构缺陷**：

> **AI 把"小说改编成剧本"当成了一次性的文本翻译任务，而不是一个需要持续记忆、可追溯、可分步审查的创作过程。**

具体来说：

- **潜台词被说穿**——因为 AI 只看到了单句对白，没有角色级的"语言风格档案"来约束它
- **千人一面**——因为"角色性格"在生成时被丢失，变成了同一套语言模型的平均输出
- **修改时全局坍塌**——因为没有"故事本体"来隔离角色设定和具体文本，改一行字就会触发全量重新随机生成
- **伏笔被吞噬**——因为摘要是一次性的，没有"因果图"来标记哪些事件是伏笔、必须在剧本中保留
- **改编非黑即白**——因为 prompt 里只有一句"忠实原著"或"发挥创意"，没有结构化的忠实度控制系统

**所以问题的解法不是"写一个更好的 prompt"，而是重新设计产品架构。**

## 三、解法：引入"轻量故事本体"（Lightweight Story Ontology）

### 3.1 为什么是 Ontology？

本体论（Ontology）的核心思想是：**把"事物是什么"和"事物怎么表现"分开。**

在剧本改编的场景中，这就意味着：

- **角色是什么**——名字、性格、目标、秘密、说话方式 → 存在一个结构化的"角色档案"里
- **角色在某一句话里怎么说**——具体的台词文本 → 存在另一个地方，但引用角色档案
- 当编剧修改台词时，**角色档案不变**，因此重新生成的对白仍然保持角色一致性
- 当编剧修改角色设定时（比如"把职业从医生改成律师"），系统知道哪些台词引用了这个角色，**只重新生成受影响的切片**，而不是全量崩塌

这就是 Ontology 解决"修改时全局坍塌"的底层逻辑。

### 3.2 产品化表达：故事圣经（Story Bible）

在产品界面里我们不叫它"本体论"——太学术了。我们叫它**故事圣经**。它包含：

| 维度 | 内容 | 解决的问题 |
|------|------|-----------|
| **角色卡** | 名字、别名、叙事角色、显性目标、隐藏欲望 | 千人一面 → 每个角色有独立的行为驱动力 |
| **声音档案** | 语言节奏、用词特征、心理防御机制 | 潜台词丧失 → LLM 按角色风格生成对白 |
| **关系边** | 角色间关系类型、当前状态、证据等级 | 关系弧线在改编中不丢失 |
| **知情状态** | 角色知道什么、不知道什么 | 避免"角色知道不该知道的信息" |
| **连续性锚点** | 称呼规范、身份事实、时间线固点 | 修改时定向传播，不全量崩塌 |
| **冲突池** | 原文中的欲望差、信息差、立场差 | 保证改编保留了原著的戏剧张力 |
| **可视化表达约束** | 内心状态→可演动作的映射规则 | 潜台词不变成直白台词 |

### 3.3 证据等级：让 AI 诚实

所有故事圣经中的内容，都标注了证据等级：

- `explicit` —— 原文明确陈述
- `inferred` —— 从原文合理推断
- `candidate` —— 可能但不确���，需人工确认

这让创作者能**审查 AI 的推断过程**，而不是面对一个"我也不知道 AI 为什么这么写"的黑箱。

## 四、架构：上下文工程 Pipeline

### 4.1 为什么不是"提示词工程"？

很多 AI 产品的工作方式是：找一个厉害的 prompt engineer，反复打磨一段 prompt，直到输出看起来不错。

我们的产品选择了一条不同的路：**上下文工程（Context Engineering）**。

区别在于：

| | 提示词工程 | 上下文工程 |
|---|---|---|
| 质量来源 | 一个 prompt 的措辞好坏 | 多层之间的结构化信息传递 |
| 错误控制 | 靠 prompt 里写"不要编造" | 靠 Schema 校验 + Reference 校验做硬约束 |
| 记忆机制 | LLM 自身上下文窗口 | 外部结构化 Story Ontology |
| 修改鲁棒性 | 改一行字全局重来 | 本体不变，文本可独立修改 |

### 4.2 Pipeline 设计：三阶段工作流

```
┌──────────────┐     ┌──────────────────┐     ┌─────────────────────┐
│  NovelReader  │ ──▶ │  StoryOntology   │ ──▶ │  ScreenplayWriter   │
│  原著解析      │     │  故事圣经+改编计划 │     │  YAML 剧本生成       │
└──────────────┘     └──────────────────┘     └─────────────────────┘
       │                       │                        │
       ▼                       ▼                        ▼
  角色候选、事件、         故事圣经、因果图、          场景、内容块、对白、
  伏笔候选、来源引用       冲突池、改编计划            潜台词、情感状态
```

**每个阶段之间，存在严格的数据契约（Data Contract）**：

- NovelReader 输出的角色和事件，StoryOntology **不能删除、不能改名、不能重排 ID**
- StoryOntology 输出的 canonical_characters 和 canonical_events，ScreenplayWriter **只能引用，不能编造新角色或新事件**
- 每个阶段都带 `source_refs`，追溯到原文的具体章节和段落

### 4.3 归一化层：不靠 Prompt 约束，靠代码保证

LLM 天然有随机性。DeepSeek 可能输出 `char_1`，下次输出 `char_baiqian`，再下次直接写中文名"白浅"。

我们的做法是：**不在 prompt 里花大量篇幅纠正格式，而是在 Pipeline 的每个阶段之后，运行一个代码层的归一化层（Normalization Layer）**：

- 自动将各种 ID 格式统一到 `char_NNN` / `event_NNN` / `foreshadow_NNN` 规范
- 根据 Story Bible 的角色查找表，自动将中文名映射回规范 ID
- 自动注册 LLM 引用了但未在角色表中声明的缺失角色
- 自动修复 Scene 中对话引用了但未在出场列表里声明的角色

**创作判断交给 Skill（LLM），结构可信交给代码。**

### 4.4 六个 Skill 的分工

| Skill | 职责 | 核心约束 |
|-------|------|---------|
| `ChapterBoundaryReader` | 识别非正文段落，定位前三章边界 | 不生成剧情内容，只标定边界 |
| `NovelReaderSkill` | 从原文抽取角色、事件、伏笔候选 | 推断内容必须标 `inferred`，不编造 |
| `StoryOntologySkill` | 构建故事圣经、因果图、改编计划 | 不生成剧本，优先可信证据 |
| `ScreenplayYamlWriterSkill` | 按 Schema 生成场景、对白、潜台词 | 每句台词含 subtext + emotional_state + action_hint |
| `DialogueDoctorSkill` | 台词润色、潜台词增强（V4 预留） | 不把潜台词说穿 |
| `ContinuityAuditorSkill` | 连续性审查、未兑现伏笔警告 | 只审查不重写 |

## 五、Schema：让 YAML 不只是排版格式

### 5.1 普通 YAML 是什么样

一般的参赛作品，YAML 可能就是：

```yaml
title: 某某小说
scenes:
  - location: 客厅
    action: 角色走进来
    dialogue: 你好
```

这只是"剧本的排版结构"。

### 5.2 我们的 Schema 设计哲学

我们的 Schema 不定义"剧本长什么样"，而是定义：

> **"AI 的改编推理过程如何被记录、追溯和审查。"**

因此顶层结构是：

```yaml
schema_version
project          # 项目元数据
source           # 原著章节引用
adaptation_config # 改编策略配置
script_structure  # 故事梗概→大纲→文学剧本 三段式
story_bible      # 轻量故事本体
core_elements    # 动作/情节/情境/主题索引
events           # 关键叙事事件
causal_graph     # 事件因果关系
foreshadowing    # 伏笔 setup → payoff 追踪
adaptation_plan  # 保留/合并/删除决策
scenes           # 剧本场景（含潜台词、情感状态、动作提示）
audit_report     # 连续性 & 潜台词审查结果
```

### 5.3 为什么一场戏要有 15 个字段？

看一个具体的 Scene 结构：

```yaml
scenes:
  - id: scene_001
    title: 雨夜重逢
    scene_heading: {sequence, location, interior_exterior, time_of_day, text}
    source_refs: [{chapter_id, paragraph_range}]    # 1. 可追溯
    dramatic_purpose: [建立主角关系, 埋下旧案伏笔]    # 2. 有目的
    location: {name, time, interior_exterior}
    characters: [char_001, char_002]
    related_events: [event_001]
    causal_links: {causes: [event_001], effects: [event_004]}  # 3. 知因果
    foreshadowing: {setups: [foreshadow_001], payoffs: []}     # 4. 管伏笔
    action: [动作描写]
    content_blocks: [action/dialogue 交替的正文块]
    dialogue:
      - character_id: char_001
        line: 你怎么会在这里？
        surface_intent: 质问对方出现原因
        subtext: 她不愿承认自己一直在等他           # 5. 潜台词不丢失
        emotional_state: anger_covering_fear
        action_hint: 她把伞柄攥得更紧              # 6. 可表演
```

**每一场戏都知道：自己从哪个原文段落来、承接了哪个事件、埋了什么伏笔、角色的潜台词是什么、情绪状态是什么、应该怎么演。**这不是一段黑箱文本，而是一份工程文件。

## 六、模型选型：为什么是 DeepSeek？

作为 AI 产品经理，大模型选型是核心能力之一。我们的选择是 **DeepSeek-v4-pro**，理由有三层：

### 6.1 上下文长度：小说改编的硬门槛

长篇小说动辄百万字。在这个尺度上，模型的上下文窗口是硬约束：

| 模型 | 上下文窗口 | 100 万字小说适配 |
|------|-----------|-----------------|
| GPT-5.4 | 256K tokens | 勉强，需分块 |
| Claude Opus 4.7 | 200K tokens | 超限，无法一次性处理单章完整上下文 |
| **DeepSeek-v4-pro** | **1M tokens** | **原生支持，前三章+楔子一次灌入** |

我们的 Pipeline 虽然只深度读取前 3 章+楔子，但每章可能数万字。1M 的上下文窗口让 NovelReader 可以一次性接收完整章节而不需要做破坏性的分段摘要——这是保留伏笔和因果链的前提。

### 6.2 中文原生理解

DeepSeek 是中文原生语境训练的模型。这不是一个"也支持中文"的模型，而是一个"以中文为第一语言"的模型。在以下场景中，这种差异是决定性的：

- 识别中文小说中的**潜台词和文化含蓄表达**（如"他把茶杯放下"可能是一个威胁信号）
- 理解**古代/仙侠/武侠的特定语域**和称谓体系（如"上神""尊者""本座"）
- 生成**符合中文剧本行业惯例**的动作描写和对白格式

### 6.3 业务适配：结构化输出 + 成本可控

DeepSeek 支持 `response_format: {"type": "json_object"}` 的结构化输出模式，与我们"每个 Skill 输出结构化 JSON"的上下文工程架构天然匹配。同时在同等 token 量下，DeepSeek 的成本显著低于 GPT-5.4，使得产品可以以一个合理的调用成本跑通完整的三阶段 Pipeline。

## 七、开发方式：Spec 驱动 Agent 开发

作为一个 AI PM 同时也是 vibe coder，这个产品的开发方式本身也是产品故事的一部分。

### 7.1 为什么用 Spec 驱动？

72 小时开发 AI 产品，最大的风险不是"写不完代码"，而是**"Agent 写的代码跟设计脱节"**。

我们的做法是：

1. **先写 Spec**：每个模块的输入输出接口、数据格式、校验规则、边界条件，全部以结构化文档定义
2. **Spec 约束 Agent**：Agent 根据 Spec 生成代码，Spec 充当了"Agent 的 prompt"
3. **验证靠自动化**：Schema Validator + Reference Validator + Audit Validator 三层校验，Agent 写的代码跑不过测试就要修

### 7.2 迭代路线与决策

| 版本 | 目标 | 关键决策 |
|------|------|---------|
| **V0** | 题目合规：3 章输入→YAML 输出 | 先保证跑通硬性要求 |
| **V1** | 可控改编：忠实度滑块 + 改编计划 | 这是"工作台感"的核心 |
| **V2** | 故事本体：Story Bible + 证据等级 | Schema 显得高级的关键 |
| **V2.1** | Pipeline 优化：adaptation_planner 并入 StoryOntology | 减少一次 LLM 调用，降低不稳定性和延迟 |
| **V2.5**（当前） | 稳定性增强：归一化层、自动角色注册、引用修复 | 让多次生成的结果在结构层面保持一致 |

## 八、当前状态与稳定性证据

### 8.1 已完成

- ✅ 多章节输入（支持楔子检测，前 3 章深度读取）
- ✅ 三阶段上下文工程 Pipeline（NovelReader → StoryOntology → ScreenplayWriter）
- ✅ 轻量故事本体（角色卡、关系边、知情状态、连续性锚点、冲突池、可视化表达约束）
- ✅ YAML 剧本生成（含潜台词、情感状态、动作提示）
- ✅ 归一化层（ID 规范、角色查找、自动注册、引用修复）
- ✅ Schema 校验 + Reference 校验
- ✅ YAML 导出（通过校验后输出）
- ✅ 中文剧本格式（场景标题、△动作、括号语气提示）
- ✅ DeepSeek 接入（JSON 结构化输出 + 3 层回退解析）

### 8.2 关于"修改不崩塌"的稳定性

虽然"编剧修改后自动更新后续"的功能还未实现（那是 V3-V5 的范围），但当前架构已经证明了核心假设：

> **因为 Story Ontology 将"角色设定"和"具体文本"分离，多次生成同一部小说的剧本时，角色性格、关系状态、关键事件的一致性显著高于端到端生成方案。**归一化层保证了结构层的一致性，Story Bible 保证了内容层的一致性。

这意味着当未来实现"局部修改→定向重生成"时，我们不需要推倒重来——架构基底已经就位。
